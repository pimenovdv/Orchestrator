# Пошаговый план реализации системы оркестрации ИИ-агентов (FastAPI + LangGraph + Temporal)

Этот документ представляет собой детализированный TODO-лист для реализации масштабируемой мультиагентной системы, основанной на архитектуре "Оркестратор-Воркер" с разделением на Control Plane и Execution Plane, согласно системной аналитике.

## Этап 1: Инициализация проекта и инфраструктура

- [ ] **1.1. Настройка репозитория и окружения**
  - Создать структуру проекта (монорепозиторий для микросервисов: Orchestrator, Player, Registry).
  - Настроить менеджер зависимостей (например, Poetry или uv) для Python.
  - Подготовить базовые конфиги линтеров (ruff, mypy, black).
- [ ] **1.2. Подготовка инфраструктуры (docker-compose)**
  - Настроить контейнер с OpenSearch (с плагином k-NN для векторного поиска).
  - Настроить контейнеры Temporal (Server, UI, PostgreSQL/Cassandra).
  - Настроить контейнеры Apache Kafka (Broker, Zookeeper/KRaft, Schema Registry).
  - Подготовить базовые Dockerfile для микросервисов на FastAPI.

## Этап 2: База агентов (Реестр и семантический поиск)

- [ ] **2.1. Моделирование данных (Pydantic)**
  - Создать строгие Pydantic-модели для JSON-манифеста агента (согласно Open Agent Spec):
    - I/O контракты (input_schema, output_schema на базе JSON Schema).
    - Промпты (system_instructions).
    - Внутренний макро-граф (nodes, edges, conditions).
    - Инструменты (built_in, rest_api, mcp, kafka).
- [ ] **2.2. Интеграция с OpenSearch**
  - Настроить подключение к OpenSearch из Python-сервиса.
  - Создать гибридный индекс агентов (BM25 для текста + k-NN для `capabilities_embedding`).
- [ ] **2.3. API Реестра (FastAPI)**
  - Реализовать CRUD-эндпоинты для управления агентами.
  - Реализовать сервис векторизации (генерация эмбеддингов для описания агентов через OpenAI или локальную модель).
  - **Контракт 2:** Реализовать эндпоинт `POST /agents_index/_search` для семантического поиска релевантных агентов по запросу.

## Этап 3: Сервис изолированного исполнения (Проигрыватель / Player)

Сервис отвечает за микро-исполнение (Micro-Graph) одного конкретного агента в строгой изоляции.

- [ ] **3.1. Базовый каркас FastAPI**
  - Создать приложение FastAPI для Player.
  - **Контракт 3:** Настроить эндпоинт `POST /api/v1/player/execute` для приема `execution_id`, `agent_manifest` и `input_context`.
- [ ] **3.2. Парсинг и валидация манифеста**
  - Реализовать валидацию `input_context` агента по его `input_schema` перед запуском.
- [ ] **3.3. Ядро исполнения на LangGraph**
  - Описать структуру состояния (State Machine) графа (TypedDict/Pydantic) для передачи контекста между узлами.
  - Создать генератор графа LangGraph на лету из JSON-манифеста агента:
    - Трансляция `nodes` в узлы LangGraph (reasoning, tool_execution, data_transformation).
    - Трансляция `edges` и `conditions` в условные переходы (conditional edges) LangGraph.
- [ ] **3.4. Интеграция LLM и Structured Outputs**
  - Настроить подключение к провайдерам (OpenAI/Anthropic).
  - Интегрировать системные промпты и роли.
  - Реализовать принудительный вывод в формате `output_schema` (используя `response_format` или инструменты).
- [ ] **3.5. Движок инструментов (Tools Engine)**
  - **Built-in:** Реализовать базовые функции (калькулятор, парсеры).
  - **REST API:** Реализовать декларативный HTTP-клиент, интерполирующий параметры из LLM в URL/Payload и фильтрующий ответ через JSONPath.
  - **MCP (Model Context Protocol):** Интегрировать MCP-клиент для вызовов `tools/list` и `tools/call` к внешним MCP-серверам.
  - **Kafka:** Реализовать продюсер/консюмер для асинхронных задач.
- [ ] **3.6. Механизмы отказоустойчивости (Resilience)**
  - Реализовать Circuit Breakers и Retries с экспоненциальной задержкой для API-инструментов.
  - Внедрить ограничения итераций (Loop Bounds) для предотвращения зацикливания.
  - Реализовать паттерн Self-Correction (Reflection): при ошибке Pydantic-валидации ответа возвращать ошибку в LLM для исправления.

## Этап 4: Сервис макро-оркестрации (Оркестратор / Control Plane)

Сервис координирует глобальный DAG (Directed Acyclic Graph) зависимостей между агентами.

- [ ] **4.1. Базовый каркас FastAPI и Temporal**
  - Создать приложение FastAPI для Orchestrator.
  - Инициализировать Temporal Client и Worker.
  - **Контракт 1:** Реализовать эндпоинт `POST /api/v1/orchestrator/dispatch` для запуска процесса.
- [ ] **4.2. Разрешение графа зависимостей (DAG)**
  - Реализовать инструмент обнаружения (AgentDiscoveryTool), запрашивающий API Реестра для поиска конечного агента.
  - Реализовать алгоритм обратной трассировки (Backward Tracing) для поиска всех зависимостей (до `dependencies == []`).
  - Реализовать проверку на циклические зависимости (Cycle Detection / Deadlock prevention).
  - Реализовать Топологическую сортировку (алгоритм Кана) для формирования волн выполнения (Execution Waves).
- [ ] **4.3. Реализация Temporal Workflow и Activities**
  - **Activity:** Вызов поиска в OpenSearch.
  - **Activity:** Построение плана выполнения (Execution Plan).
  - **Activity:** Диспетчеризация — вызов Проигрывателя (`POST /api/v1/player/execute`) для каждого агента в волне.
  - **Workflow:** Управление общим процессом: параллельный запуск агентов из одной волны, сборка результатов в State Store, маршрутизация контекста от завершенного агента к следующему зависимому (сопоставляя с его `input_schema`).
- [ ] **4.4. Интеграция Kafka-триггеров (опционально)**
  - Настроить Temporal Worker или отдельный сервис для прослушивания топиков Kafka и старта оркестрации по входящему событию.

## Этап 5: Тестирование и стабилизация

- [ ] **5.1. Unit-тестирование**
  - Покрытие тестами Pydantic-моделей манифестов.
  - Покрытие тестами алгоритма топологической сортировки и поиска циклов.
- [ ] **5.2. Интеграционное тестирование**
  - Тестирование семантического поиска в OpenSearch.
  - Запуск Проигрывателя с моковым JSON-манифестом (REST-вызов, простой reasoning).
- [ ] **5.3. E2E тестирование**
  - Проверка сквозного сценария: `Запрос -> Оркестратор -> Поиск -> Построение DAG -> Temporal Workflow -> Выполнение волн в LangGraph -> Итоговый валидированный JSON-ответ`.
- [ ] **5.4. Валидация JSON-схем для фронтенда**
  - Убедиться, что все конфигурации можно экспортировать в виде JSON Schema для последующего подключения No-Code визуализаторов (например, React Flow, JSON Forms).
