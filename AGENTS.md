# Инструкции для ИИ-агентов (AI Agents)

Этот файл содержит ключевые инструкции, архитектурные особенности и договоренности для ИИ-агентов, работающих с кодовой базой данного проекта (Оркестратор ИИ-агентов).

## 1. Архитектура и стек
- **Язык и фреймворки:** Python (>=3.12), FastAPI (шлюз), Temporal (оркестрация рабочих процессов), LangGraph (микро-граф внутри агента, для симуляции выполнения), OpenSearch (реестр агентов и семантический поиск), Pydantic (схемы и валидация).
- **Менеджер пакетов и запуска:** `uv`. Все скрипты, тесты, линтеры запускаются через него (например: `uv run pytest`, `uv run mypy .`).
- **Инфраструктура локальной разработки:** Docker Compose V2 (`docker compose`). Включает Temporal, OpenSearch и Redis.

## 2. Организация кода
- Весь исходный код приложения находится в директории `src/app/`.
- Импорты в проекте должны использовать префикс `app.` (например: `from app.models.agent import ...`), а **НЕ** `src.app.`. Это критично для совместимости с `mypy`.
- Тесты лежат в `tests/`, вспомогательные скрипты — в `scripts/`.

## 3. Код-стайл и проверки
- **Mypy:** Настроен строго (`strict=true`, `[no-untyped-def]`). Обязательно указывать типы возвращаемых значений для всех функций, *даже для тестов* (например, `def test_smth() -> None:`). При запуске может потребоваться указать `PYTHONPATH=src uv run mypy .`.
- **Ruff и Black:** Используются для линтинга и форматирования кода. Форматируйте код после внесения изменений.

## 4. Особенности работы с Temporal
- Workflows (рабочие процессы) определяются декоратором `@workflow.defn` и лежат в `src/app/temporal/workflows.py`.
- Activities (активности) определяются декоратором `@activity.defn` и лежат в `src/app/temporal/activities.py`.
- Параллельное выполнение Activities внутри Workflows (например, запуск волн графа агентов - DAG waves) реализуется с помощью `asyncio.gather`.
- При вызове Workflows из FastAPI (в `endpoints.py`), используйте `client.start_workflow` (асинхронный запуск без блокировки HTTP-ответа), а не `client.execute_workflow`.
- Обработка отсутствующих запущенных Workflows: ловите `temporalio.service.RPCError`, проверяя текст ошибки на предмет "not found" (а не `temporalio.exceptions.RPCError`).

## 5. Особенности тестирования
- **Pytest:** Запускается через `uv run pytest` (директория `src` прописана в `pythonpath` в `pyproject.toml`).
- **FastAPI:** Используйте `httpx.AsyncClient(transport=ASGITransport(app=app), base_url="http://test")` для тестирования эндпоинтов без поднятия сервера.
- **Моки Temporal:**
  - При мокировании `temporalio.client.Client`, метод `get_workflow_handle` является **синхронным**, в то время как методы самого хендла (например, `describe()`) — **асинхронные**. Возвращайте из синхронного мока хендла объект `AsyncMock`.
  - Для юнит-тестирования Workflows и Activities используйте `temporalio.testing.WorkflowEnvironment` и `Worker`.
- **Внешние HTTP-запросы:** Мокируются с помощью библиотеки `respx`.
- **LangGraph (E2E тесты):** Используется внутри замоканных Activities для симуляции выполнения агента и возврата тестовых данных.

## 6. Взаимодействие с OpenSearch
- Используется асинхронный клиент `AsyncOpenSearch` из пакета `opensearch-py[async]`.
- При отсутствии документа (например, манифеста агента) нужно явно обрабатывать исключение `opensearchpy.exceptions.NotFoundError`.

## 7. Контракты Pydantic
- Модели Pydantic (такие как `ExecuteRequest`, `ExecuteResponse`, `Tool` и т.д.) определяют строгий контракт связи с внешним сервисом `runner`/`player`.
- Они должны быть функционально разделены на модули (`api.py`, `tools.py`, `graph.py`, `manifest.py`), чтобы избежать дублирования кода.
- Использовать Literal/Optional, если это требуется контрактом, вместо Enum/Union.
