# Структура JSON Агента

В основе архитектуры мультиагентной системы лежит декларативная JSON-структура агента (концепция «конфигурация вместо кода»). Это строгий контракт, описывающий интерфейсы ввода-вывода, системные инструкции, доступные инструменты и внутреннюю машину состояний (граф переходов).

Каждый документ агента в индексе (например, в OpenSearch/Elasticsearch) включает в себя метаданные и сам `manifest` (полная JSON-структура).

## Метаданные (AgentIndexDocument)
- **agent_id**: Уникальный детерминированный идентификатор агента (например, `finance_auditor_v1`). Тип: `keyword`.
- **name**: Человекочитаемое имя агента для визуализации и лексического поиска. Тип: `text`.
- **description**: Подробное описание компетенций, бизнес-логики и ограничений агента. Тип: `text`.
- **capabilities_embedding**: Плотный вектор (например, 1536 измерений для OpenAI), сгенерированный на основе `description`, используемый для семантического поиска. Тип: `knn_vector` (или `dense_vector` в ElasticSearch).
- **dependencies**: Массив идентификаторов (`agent_id`) других агентов, от вывода которых зависит данный узел. Тип: `keyword (array)`.
- **manifest**: Полная декларативная JSON-структура агента (описана ниже). Тип: `object`.

## Manifest (JSON-структура агента)

`manifest` содержит следующие основные секции:

### 1. Контракты данных (I/O Contracts)
- **input_schema**: JSON Schema входящих данных. Строго типизирует ожидаемый контекст перед вызовом агента.
- **output_schema**: JSON Schema исходящих данных. Гарантирует формат возвращаемых агентом данных (например, через `response_format` в LLM).

### 2. Системные промпты и ролевые модели (prompts)
- **system_instructions**: Системные инструкции, описывающие роль агента, формат рассуждений (например, требование использовать Chain-of-Thought) и стратегии обработки ошибок.
- **guardrails**: (Опционально) Ограничения безопасности.

### 3. Инструменты (tools)
Массив доступных агенту инструментов. Каждый инструмент описывается объектом:
- **name**: Имя инструмента.
- **type**: Тип интеграции (`rest_api`, `mcp_server`, `kafka`, или `builtin`).
- **description**: (Опционально) Описание инструмента для LLM.
- **rest_api_config**: (Опционально) Конфигурация для `rest_api`:
  - `method`: HTTP метод (GET, POST, PATCH и др.).
  - `base_url`: Базовый URL.
  - `headers`: Заголовки (опционально).
  - `authentication`: Настройки аутентификации (опционально).
  - `parameters_schema`: JSON Schema параметров, которые должна сгенерировать LLM (query parameters или payload).
- **mcp_server_config**: (Опционально) Конфигурация для `mcp_server`:
  - `server_url`: URL сервера MCP.
- **kafka_config**: (Опционально) Конфигурация для интеграции с Apache Kafka:
  - `topic`: Топик Kafka для подписки или публикации.
  - `bootstrap_servers`: Адреса брокеров Kafka.
  - `message_schema`: (Опционально) JSON Schema отправляемого/получаемого сообщения.
- **builtin_config**: (Опционально) Конфигурация для встроенных платформенных инструментов (например, AgentDiscoveryTool):
  - `function_name`: Идентификатор встроенной функции.

### 4. Внутренняя машина состояний (graph / Micro-Graph)
Граф переходов, который определяет изолированные шаги (узлы) и условные переходы (ребра). Позволяет реализовать сложные циклические операции (например, Reflection) без написания кода.
- **nodes**: Массив узлов (изолированных этапов).
  - `id`: Уникальный ID узла.
  - `type`: Тип узла (например, `reasoning`, `tool_execution`, `data_transformation`).
  - `tool_name`: Имя инструмента (для типа `tool_execution`).
  - `description`: (Опционально) Описание шага.
- **edges**: Массив ребер (направленных связей между узлами).
  - `source`: ID исходного узла.
  - `target`: ID целевого узла.
  - `condition`: (Опционально) Логическое выражение для перехода (условие маршрутизации / Skip Logic).
    - `condition_expression`: Строка с логическим выражением.

### 5. Ограничения выполнения (execution_limits)
- **max_tokens**: Максимальное количество токенов, разрешенное для использования агентом (например, 8000).
- **timeout_ms**: Таймаут выполнения в миллисекундах (например, 60000).

---
*Пример запроса к Player Execution:*
```json
{
  "execution_id": "exec-5555-uuid",
  "agent_manifest": { ... Полная JSON-структура агента ... },
  "input_context": {
    "incident_id": "#404",
    "raw_logs": "[2026-03-25 18:50:00] ERROR Database connection failed"
  },
  "execution_limits": {
    "max_tokens": 8000,
    "timeout_ms": 60000
  }
}
```
